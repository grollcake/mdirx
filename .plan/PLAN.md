# MdirX 기능 계획 문서

개별 기능별 요구사항, 아키텍처, 구현 체크리스트를 정리하는 디렉토리입니다.

---

## 계획 우선 규칙

1. **항상 계획부터 세운다.** 구현·리팩터·버그 수정 포함, 작업 착수 전에 이 디렉터리에 계획 문서를 둔다.
2. **사용자 요구를 정제해 남긴다.** 계획 작성의 출발점이 된 요청·대화를 **원래 의도가 훼손되지 않게** 다듬어 `사용자 요구 (정제)` 절에 기록한다(아래 템플릿).
3. **계획에는 수도 코드를 넣는다.** 제어 흐름·상태 전환·알고리즘 스텝을 자연어·기호로 적어, 코드와 1:1 대응이 되도록 한다.
4. **통과 조건을 명시한다.** “완료”의 판정 기준을 측정 가능하게 쓴다 (테스트 항목, 수동 검증 시나리오, 성능/UX 기준 등).

계획 문서는 **`.plan` 디렉터리에서만** 관리한다. (루트 `PLAN.md` 등과 연동할 때는 이 규칙을 따르는 개별 계획 파일로 쪼개서 링크한다.)

---

## 계획 문서 파일명

```
{mmdd}-{hhmm}-{요구사항요약}.{상태}.md
```

| 부분 | 설명 |
|------|------|
| `mmdd` | 월·일, 각 2자리 zero-pad (예: 5월 16일 → `0516`) |
| `hhmm` | 24시간 시·분, 2자리씩 (예: 오후 2시 30분 → `1430`) |
| `요구사항요약` | **최대 20자**, 공백·특수문자는 가급적 피하고 하이픈으로 단어 구분 |
| `상태` | 정확히 **`todo` \| `doing` \| `done`** 셋만 사용 (소문자·영문 고정) |

**`done` 조건:** 자동 전이 없음. **사용자 검증**을 거쳐 **사용자 OK**를 받은 뒤에만 파일명·문서 상의 상태를 `done`으로 바꾼다. OK 이후 **새로 배운 점·헤매다 해결한 점·반복 가능한 함정**은 [docs/learnings/](docs/learnings/) 에 **후보로 제안**하고, **사용자가 승인한 것만** 기록으로 추가한다([`AGENTS.md`](../AGENTS.md)).

**예:** `0516-1430-dual-pane-tabs.todo.md`

계획 문서는 **하위 폴더 없이** `.plan/` 루트에 둔다. 이름을 바꿀 때(`todo`→`doing`→`done`)에도 같은 디렉터리에 둔 채 파일명만 갱신한다.

## 작성 가이드

각 계획 문서는 아래 구조를 따른다.

````markdown
# 기능명 (또는 작업 제목)

## 사용자 요구 (정제)
- 이 문서를 만들게 된 **요청의 핵심**을 사용자 관점으로 1~3문장(또는 불릿).
- 필요 시 채팅·이슈 **원문 인용**(짧게) + 그 아래 **의도 유지·모호함 제거**한 정제 문장.
- 구현 디테일은 쓰지 않고, **왜 이 계획이 생겼는지**만 남긴다.

## 개요
한 문단으로 목적·범위

## 요구사항
- UI 요구사항
- 동작 요구사항
- 성능/제약

## 수도 코드
```
여기에 흐름·분기·루프·상태를 수도 코드로 기술
```

## 아키텍처
- 관련 View / ViewModel
- 데이터 흐름
- 외부 의존성

## 통과 조건
- [ ] 측정 가능한 완료 기준 1 (테스트명 또는 검증 절차)
- [ ] 측정 가능한 완료 기준 2
- 엣지·에러 시 기대 동작

## 구현 체크리스트
- [ ] 작업 1
- [ ] 작업 2

## 테스트 케이스
- 정상 케이스
- 엣지 케이스
- 에러 케이스
````

**상태 전이:** `todo`(착수 전) → `doing`(진행 중) → `done`. `done`은 **사용자 검증**을 마치고 **사용자 OK**를 받은 뒤에만 적용한다(파일명 `{상태}`와 본문 기록 일치).

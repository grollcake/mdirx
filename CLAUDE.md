# 에이전트 작업 규칙

## 계획 문서 우선

사용자 요청에 **계획을 세우거나 작성하라는 취지**가 포함되면(예: 영어 `plan`, `planning`, `write a plan`, `start with a plan` / 한국어 `계획`, `계획부터`, `계획 세워`, `계획 작성`, `먼저 계획`, `작성해`가 **계획 문서** 맥락일 때) **코드 변경·구현·리팩터에 앞서** [`.plan/PLAN.md`](.plan/PLAN.md)를 따른다.

1. **항상 먼저** [docs/learnings/](docs/learnings/) 를 살핀다(`README.md` 목록·해당 **영역** 디렉터리). 과거 실수·올바른 해결과 충돌·중복이 없도록 계획(특히 아키텍처·통과 조건)에 반영한다. 관련 기록이 없으면 짧게라도 “learnings에서 해당 주제 없음” 정도로 스킵 이유를 남겨도 된다.
2. `.plan/` 루트에 계획 파일을 만든다(하위 폴더 없음).
3. 파일명·본문 구조(특히 **사용자 요구 (정제)**)·수도 코드·통과 조건·상태(`todo` / `doing` / `done`) 규칙은 전부 `.plan/PLAN.md`에 맞춘다. 최초 작성은 보통 `*.todo.md`.
4. 사용자가 **계획을 건너뛰고 바로 구현**하라고 명시적으로 말한 경우에만 이 순서를 생략한다.

## 설계·계획 범위 질문 시: `TODO` / `PLAN` / `PRD` 검토 후 계획 문서 제안

사용자가 **무엇을 설계·계획해야 하는지**, **앞으로 어떤 계획 문서를 작성할지**, **기능·스펙을 어떻게 쪼개 계획할지** 등을 물을 때(순수 **구현 실행 순서**만 묻는 경우는 아래 **할 일·구현 질문** 절에 가깝다) 다음을 따른다.

1. **[`TODO.md`](TODO.md)** · **[`PLAN.md`](PLAN.md)** · **[`docs/PRD.md`](docs/PRD.md)** 를 **먼저** 읽어 제품 방향·마일스톤·백로그·비범위를 파악한다.
2. **[`.plan/STATUS.md`](.plan/STATUS.md)** 및 `.plan/*.md` 와 맞춰 본다. 이미 `todo`/`doing`으로 다루는 주제와 **아직 `.plan`으로 쪼개지 않은 공백**을 구분한다.
3. 공백·다음 단계에 대해 **[`.plan/PLAN.md`](.plan/PLAN.md)** 규격을 따르는 **신규 `*.todo.md` 작성**을 제안한다(제안 범위: 파일명·첫 `#` 제목·핵심 통과 조건·`STATUS` 행 추가 필요 여부). 구현에 들어가기 전 사용자가 초안을 확정할 수 있게 한다.
4. 위 세 문서가 없거나 비어 있고 `.plan`과도 연결되지 않으면 그 사실을 밝히고, `PRD`/`PLAN`/`TODO` 정비 또는 요구 보강을 먼저 제안한다.

## 할 일·구현 질문 시: `todo` 계획 최우선

사용자가 **무엇을 해야 하는지**, **다음에 무엇을 구현할지**, **할 일·백로그·우선순위** 등을 물으면([`TODO.md`](TODO.md)나 [`PLAN.md`](PLAN.md) 전체를 먼저 훑기 전에) **최우선**으로 아래를 제시한다.

1. **[`.plan/STATUS.md`](.plan/STATUS.md)** 표에서 **상태가 `todo`** 인 행을 모두 나열한다(계획 문서 링크·시작일시 등 표에 있는 그대로).
2. **`.plan/*.todo.md`** 를 확인하여, `STATUS` 와 대응하는 파일의 **제목(첫 `#`)**·통과 조건·체크리스트를 **짧게** 인용하거나 링크한다. (`todo` 계획이 `STATUS` 에 아직 없으면 파일명과 함께 **표 갱신이 필요할 수 있음**을 한 줄로 짚는다.)
3. **`todo` 가 없으면** 그 사실을 먼저 밝히고, 이어 **`doing`** 계획 → 필요 시 **`done`** 최근 요약 순으로 안내한다. 저장소에 정의되지 않은 작업을 임의로 만들어 우선순위에 넣지 않는다.

이후에야 [`TODO.md`](TODO.md)·[`PLAN.md`](PLAN.md)·[`docs/PRD.md`](docs/PRD.md) 등으로 시야를 넓혀도 된다.

## 문제 발생 시: 추정 수정 금지, 증거 우선

버그·빌드 실패·CI 실패·회귀 등 **문제가 생겼을 때** 다음을 지킨다.

1. **추정만으로 코드를 고치지 않는다.** “아마 이거일 것” 수준의 가설만으로 대규모 변경·여러 파일 동시 수정·설정 뭉뚱그림 변경을 하지 않는다.
2. **수정에 앞서 증거를 수집한다.** 예: 재현 절차, 전체/관련 **에러 메시지·스택 트레이스**, `xcodebuild`/테스트/린트 **출력 발췌**, 문제가 난 **커밋·파일·라인**, 설정 diff, (가능하면) 최소 재현 조건. 필요하면 로그 추가·한 단계씩 좁히는 관측을 먼저 한다.
3. **가설은 증거와 연결한 뒤** 가장 작은 범위로 검증한다. 증거 없이 가설만 바꿔 가며 시행착오하는 것을 피하고, 한 번에 하나의 원인 후보만 분리해 확인한다.

증거가 부족하면 **추가로 어떤 정보를 모을지**를 먼저 밝히고, 수집 후에 수정한다.

## 동일 작업 3회 실패 시: 중단·보고·처분 대기

같은 기능·요구사항을 두고 **구현 시도가 3회 이상 실패** 했다면 **즉시 중단** 한다. 같은 곳을 반복해서 헛도는 것을 막기 위함이다.

1. **카운트 기준** — 같은 파일·컴포넌트·통과 조건을 향한 시도. 다음을 모두 1회 실패로 본다.
   - 빌드 실패(컴파일 오류) 후 재시도.
   - 자동 테스트 실패(`xcodebuild test` 또는 단위·UI 테스트) 후 재시도.
   - 사용자 검토에서 "다시" 또는 동일 결함 지적을 받은 후 재시도.
   - 가설을 명확히 바꾸어 접근을 갈아탔더라도 **누적 카운트** 에 포함(다른 접근이 효과적이라는 증거가 사용자 승인 없이 정해지지 않기 때문).
2. **3회 도달 직후 행동**
   - 추가 구현 시도를 **즉시 멈춘다.** 코드 추가 수정·테스트 재실행·다른 가설로의 자동 전환을 하지 않는다.
   - 사용자에게 다음을 한 번에 보고:
     - ① **무엇** 을 구현하려 했는지(계획 문서 링크 + 통과 조건 요약).
     - ② **세 번의 시도** 각각이 어디서·어떻게 실패했는지(에러 메시지·테스트 출력·사용자 코멘트 인용).
     - ③ **남은 후보 가설/대안** 2~3개 — 각각의 위험·범위·예상 비용.
     - ④ **사용자 처분 요청**: 가설 선택, 범위 변경, 다른 접근 지시, 또는 계획 자체의 재정의 중 무엇을 원하는지.
3. **재개 조건** — 사용자가 명시적으로 처분을 내린 뒤에만 카운트를 리셋하고 재시도한다. "한 번 더 해봐" 도 사용자가 직접 말해야 한다. 자동 재시도는 금지.

이 규칙은 [`.plan/`](.plan/) 의 통과 조건과 결합되어 작동한다. 통과 조건이 모호해 시도마다 합격 판정이 흔들리는 경우는 3회 실패 보고에 **"통과 조건을 더 구체화 필요"** 를 후보로 함께 적는다.

## 과도한 수정 금지, 요청 범위·기존 구조 존중

현재 저장소의 디렉터리·모듈 경계·문서 규격 등 **의도된 구조를 전제로** 한다.

1. **요청받은 부분만** 정교하게 고친다. 같은 턴이라도 “정리 겸” 리팩터·스타일 통일·넓은 범위의 이름 변경·요청과 무관한 파일 건드리기를 하지 않는다.
2. **기존 코드의 패턴·맥락**을 따른다. 네이밍, 분리 방식, 주석/문서 톤, import·레이어 관례를 흐트러뜨리지 않고, 추가분은 그 연장선에서 최소로 맞춘다.
3. 기존 구조나 설계가 **이상해 보이거나** 개선 여지가 있어도, **임의로 넓게 고치지 않는다.** 먼저 사용자에게 무엇이 어색한지·어떤 확대 수정이 필요할지 **짧게 보고**하고, **승인된 뒤** 범위를 넓혀 수정한다.

## 완료 승인 → learnings

작업 **완료에 대한 사용자 OK** 이후, 아래에 해당하는 **learnings 후보**를 사용자에게 제안한다(후보마다 한 줄 요약·넣을 영역 디렉터리 정도).

- **새로 배운 것**
- **헤매다가 겨우 해결한 것**(실수 → 올바른 해결)
- **동일하게 반복될 수 있는 것**(다음에 같은 함정)

**사용자가 추가를 승인한 후보에 한해** [docs/learnings/](docs/learnings/) 에 문서를 쓴다. 승인 없이 저장소에 추가하지 않는다. 이미 비슷한 문서가 있으면 후보 설명에 밝히고, 갱신·링크안을 제안할 수 있다.

`.plan` 계획·[`STATUS.md`](.plan/STATUS.md)를 `done`으로 맞추는 대화와 **같은 흐름에서 후보 제안**까지 하는 것을 원칙으로 한다. 문서 반영은 사용자 승인 뒤에 진행한다.

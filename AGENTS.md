# 에이전트 작업 규칙

## 계획 문서 우선

사용자 요청에 **계획을 세우거나 작성하라는 취지**가 포함되면(예: 영어 `plan`, `planning`, `write a plan`, `start with a plan` / 한국어 `계획`, `계획부터`, `계획 세워`, `계획 작성`, `먼저 계획`, `작성해`가 **계획 문서** 맥락일 때) **코드 변경·구현·리팩터에 앞서** [`.plan/README.md`](.plan/README.md)를 따른다.

1. **항상 먼저** [docs/learnings/](docs/learnings/) 를 살핀다(`README.md` 목록·해당 **영역** 디렉터리). 과거 실수·올바른 해결과 충돌·중복이 없도록 계획(특히 아키텍처·통과 조건)에 반영한다. 관련 기록이 없으면 짧게라도 “learnings에서 해당 주제 없음” 정도로 스킵 이유를 남겨도 된다.
2. `.plan/` 루트에 계획 파일을 만든다(하위 폴더 없음).
3. 파일명·본문 구조(특히 **사용자 요구 (정제)**)·수도 코드·통과 조건·상태(`todo` / `doing` / `done`) 규칙은 전부 `.plan/README.md`에 맞춘다. 최초 작성은 보통 `*.todo.md`.
4. 사용자가 **계획을 건너뛰고 바로 구현**하라고 명시적으로 말한 경우에만 이 순서를 생략한다.

## 완료 승인 → learnings

작업 **완료에 대한 사용자 OK** 이후, 아래에 해당하는 **learnings 후보**를 사용자에게 제안한다(후보마다 한 줄 요약·넣을 영역 디렉터리 정도).

- **새로 배운 것**
- **헤매다가 겨우 해결한 것**(실수 → 올바른 해결)
- **동일하게 반복될 수 있는 것**(다음에 같은 함정)

**사용자가 추가를 승인한 후보에 한해** [docs/learnings/](docs/learnings/) 에 문서를 쓴다. 승인 없이 저장소에 추가하지 않는다. 이미 비슷한 문서가 있으면 후보 설명에 밝히고, 갱신·링크안을 제안할 수 있다.

`.plan` 계획·[`STATUS.md`](.plan/STATUS.md)를 `done`으로 맞추는 대화와 **같은 흐름에서 후보 제안**까지 하는 것을 원칙으로 한다. 문서 반영은 사용자 승인 뒤에 진행한다.
